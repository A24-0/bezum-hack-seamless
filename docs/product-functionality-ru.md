# Seamless / Bezum Platform — назначение и функционал

Платформа объединяет **управление проектом**, **задачи**, **документы с версиями**, **встречи с согласованием времени и саммари**, **связь с GitLab (CI/CD на уровне PR и релизов)** и **уведомления**. Ниже — как это должно работать с точки зрения продукта и зачем в требованиях фигурирует CI/CD.

---

## Роли и доступ

- **Менеджер** — проекты, спринты, приоритеты, документы, встречи, привязка PR к задачам.
- **Разработчик** — задачи, документы, встречи, участие в согласовании слотов, работа с PR в интерфейсе.
- **Заказчик** — просмотр согласованного scope, встреч, документов (в демо — ограничения по ролям на отдельные действия).

Доступ к API проверяется по JWT; к данным проекта — по членству в проекте.

---

## Проекты

- Список проектов, карточка с прогрессом и метаданными (спринты, участники, ссылка на репозиторий).
- У проекта может быть URL **GitLab** — это точка входа для смысловой связи «код ↔ задачи».

---

## Спринты (epochs)

- Планирование итераций: цели, даты начала/окончания, статус (планирование / активен / завершён и т.д.).
- **Валидация дат:** дата окончания не может быть раньше даты начала (проверка на клиенте и на сервере при создании; при обновлении — проверка на сервере после применения полей).

---

## Задачи и Kanban

- Задачи в колонках статусов: бэклог, к выполнению, в работе, на ревью, готово (и при необходимости «нужна информация» в данных).
- Смена статуса обновляет задачу на сервере; связано с уведомлениями и обзором проекта.

---

## Документы

- Создание документов, **версии** с кратким описанием изменений.
- Связь с **задачами** (ручная и автоматическая по упоминаниям).
- Утверждение документа (статусы черновик / на проверке / утверждён и т.д. — по модели данных).
- Контекст встреч при необходимости (в т.ч. подтягивание саммари).

---

## Встречи

- Создание встречи, опционально **дата/время**; при указании времени встреча переводится в статус «запланировано».
- **Валидация даты/времени:** при создании встречи и при предложении слотов проверяется корректность значения; на сервере — разбор ISO без падения на мусорных строках.
- Для встреч в режиме согласования: **предложение слотов**, голосование, **фиксация времени** (в т.ч. автовыбор лучшего слота — по логике бэкенда).
- **Транскрипт** и **саммаризация** (если подключена служба) — для фиксации договорённостей и связи с документами.

---

## Уведомления

- Список уведомлений, отметка прочитанным, счётчик непрочитанных в шапке.
- Опционально WebSocket для push в реальном времени (если поднят тот же хост и токен передан).

---

## Карта связей (интеграции)

- Обзорное представление: сколько сущностей в проекте и как задачи связаны с PR и встречами (эвристики/явные связи).

---

## CI/CD в продукте: зачем и что делает

В требованиях **CI/CD** здесь означает не «настроить пайплайн внутри приложения», а **сквозную прозрачность разработки** для менеджера и заказчика:

1. **Pull Request (Merge Request) из GitLab** отображаются в проекте как сущности со статусом (открыт / слит / закрыт), ветками source/target и ссылкой на GitLab.
2. **Привязка PR к задаче** — вручную в UI: выбранная задача отражает «какой тикет закрывает этот MR». Это связывает план (Kanban) с фактом кода.
3. **Релизы** привязаны к **спринтам** (epoch): фиксируется имя релиза, тег версии, описание — как артефакт поставки итерации.
4. **Синхронизация** — кнопка «Синхронизировать» вызывает заглушку `POST /projects/{id}/sync`; в документации API указано, что **реальное обновление** ожидается от **GitLab webhooks** (`/webhooks/gitlab`), когда интеграция будет полностью развёрнута. То есть CI/CD в продукте = **видимость MR + релизы + связь с задачами**, а не замена GitLab CI.

Итог: **CI/CD блок в приложении** отвечает на вопросы: «какие MR идут в репозиторий проекта», «к какой задаче относится MR», «какой релиз из какого спринта», без дублирования самого пайплайна сборки.

---

## Тема оформления (день / ночь)

- Переключатель в шапке меняет класс `dark` на `<html>` и сохраняет выбор в `localStorage` (`seamless-theme`).
- Стили карточек, полей ввода и фона поддерживают светлую и тёмную тему.

---

*Документ отражает целевое поведение; отдельные эндпоинты могут быть расширены (webhooks, внешние интеграции) без смены общей модели.*
